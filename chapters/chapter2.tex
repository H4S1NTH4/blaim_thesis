\section{Research Methodology}
\label{sec:methodology}

This study adopts a \textbf{design science research} orientation with a quantitative event-analytics implementation layer. The methodological flow consists of problem identification, artifact design, iterative implementation, and scenario-based evaluation. The primary artifact is a deployable behavior learning analytics module embedded in the GradeLoop platform.

The methodology combines:
\begin{itemize}
	\item \textbf{Literature-driven design inputs} from keystroke biometrics and computing-education analytics \citep{acien2022typenet, senarath2023reevaluating, arakawa2021situ};
	\item \textbf{Implementation-driven validation} using system traces, API-level tests, and functional acceptance criteria;
	\item \textbf{Interpretability-focused output design} for instructors and academic integrity workflows.
\end{itemize}

\subsection{Research design}
\label{subsec:design}

The research design is structured in five phases:

\begin{enumerate}
	\item \textbf{Requirement Engineering:} Define educational and security requirements for continuous identity assurance and learner-process visibility.
	\item \textbf{Data Model Definition:} Formalize keystroke event schema with timestamps, dwell time, flight time, key categories, and action metadata.
	\item \textbf{Authentication Pipeline Construction:} Build a TypeNet-style embedding workflow for enrollment and continuous verification.
	\item \textbf{Behavior Analytics Construction:} Implement metrics for friction points, pause patterns, deletion dynamics, cognitive load trends, and authenticity indicators.
	\item \textbf{System Integration and Evaluation:} Expose functionality through FastAPI endpoints and assess behavior under representative coding-session scenarios.
\end{enumerate}

This design treats authentication and analytics as coupled components sharing one event substrate, reducing data duplication and improving interpretability consistency.

\subsection{Data collection}
\label{subsec:data}

Data collection is event-based and privacy-aware. The system captures interaction telemetry from coding sessions, including key timing and action annotations, while minimizing retention of sensitive textual content. Each event contains timestamp, key identity (or category), dwell time, flight time, and action type (e.g., type, delete, paste).

Enrollment data is collected in phased tasks to capture stable and context-varying typing behavior:
\begin{itemize}
	\item \textbf{Baseline phase:} low-stress credential-style typing,
	\item \textbf{Transcription phase:} controlled code copying to observe pure motor rhythm,
	\item \textbf{Optional stress/cognitive phases:} timed and logic-heavy tasks to capture behavioral shifts.
\end{itemize}

These phases support robust template formation and mitigate overfitting to a single typing context. For continuous authentication, incoming sessions are segmented into fixed-length sequences for rolling verification and risk monitoring.

%% ── 2.2 Commercialization Aspects ────────────────────────────
\section{Commercialization Aspects of the Product}
\label{sec:commercial}

The proposed module is commercially viable as a \textbf{trust and analytics extension} for digital assessment platforms.

\textbf{Target market.} Universities, coding bootcamps, online examination providers, and LMS vendors that require both integrity controls and actionable learning insights.

\textbf{Value proposition.}
\begin{itemize}
	\item Continuous identity confidence rather than one-time login checks,
	\item Explainable risk signals (paste anomalies, unrealistic burst patterns, abrupt rhythm shifts),
	\item Pedagogical outputs (struggle zones, troubleshooting style, engagement profiles).
\end{itemize}

\textbf{Deployment model.} The module can be offered as a containerized microservice with REST APIs, allowing SaaS and on-premise deployment.

\textbf{Revenue options.} Institutional licensing per active learner cohort, usage-based API billing, and analytics-tier subscriptions for instructor dashboards.

\textbf{Risk and compliance considerations.} Adoption requires transparent data governance, student consent mechanisms, configurable retention policies, and clear human-in-the-loop review for high-risk flags.

%% ── 2.3 Testing and Implementation ───────────────────────────
\section{Testing and Implementation}
\label{sec:testing}

Implementation follows a service-oriented architecture with Python/FastAPI for keystroke intelligence, TypeNet-style deep embeddings for biometric matching, and web dashboard integration for instructor-facing analytics \citep{gradeloop2026docs}. Key implementation artifacts include event ingestion, feature extraction, template enrollment, sequence verification, and report generation.

The testing strategy includes:
\begin{itemize}
	\item \textbf{Unit-level checks} for feature extraction and scoring computations,
	\item \textbf{API contract tests} for enrollment, verification, and behavioral analysis endpoints,
	\item \textbf{Scenario-based integration tests} using representative session traces to evaluate end-to-end outputs,
	\item \textbf{Robustness checks} for sparse data, abnormal typing bursts, and high paste incidence.
\end{itemize}

\subsection{System architecture}
\label{subsec:architecture}

The architecture consists of three interacting layers:

\begin{enumerate}
	\item \textbf{Capture layer:} Frontend instrumentation records keystroke events during coding.
	\item \textbf{Intelligence layer:} Keystroke-service microservice computes authentication and behavior analytics.
	\item \textbf{Consumption layer:} API gateway and instructor dashboards consume risk and pedagogical outputs.
\end{enumerate}

For sequence modeling, each keystroke vector is represented as:

\begin{equation}
\mathbf{x}_t = [HL_t, IL_t, PL_t, RL_t, K_t]
\end{equation}

where $HL$ is hold latency, $IL$ is inter-key latency, $PL$ is press latency, $RL$ is release latency, and $K$ is normalized key code.

Continuous verification uses cosine similarity between current embedding $\mathbf{e}_t$ and enrolled template $\mathbf{u}$:

\begin{equation}
\mathrm{sim}(\mathbf{e}_t,\mathbf{u}) = \frac{\mathbf{e}_t \cdot \mathbf{u}}{\|\mathbf{e}_t\|\,\|\mathbf{u}\|}
\end{equation}

The session-level risk can be expressed as:

\begin{equation}
\mathrm{risk}_t = 1 - \mathrm{sim}(\mathbf{e}_t,\mathbf{u})
\end{equation}

Behavior scores are computed from interpretable rule sets. For example, the overall process quality score is the arithmetic mean of active problem solving, learning depth, authenticity, and engagement components.
